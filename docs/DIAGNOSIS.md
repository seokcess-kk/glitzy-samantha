# Samantha 진단 통합본 — 병원 실무자 · 대행사 관리자 업무 개선

> 작성: 2026-07-24 · 방식: 코드 미수정 진단(DB migrations + 실제 컬럼 → API 89개 라우트 → 프론트 추적). 근거는 `파일:라인`으로 명시.

## 분석 전제 — 모집단은 "유료 광고 리드"

이 시스템의 신규 상담 리드는 **당사가 운영·연동한 유료 광고 매체를 통해 유입된 리드만**을 대상으로 한다. 오가닉·직접·전화·지인 소개·자연 검색/SNS 등 비광고 유입을 별도로 등록·분석·비교하는 시스템이 아니다. 따라서 본 문서 전반에 다음 원칙을 적용한다.

- 마케팅 성과 모집단 = **운영 광고 매체로 생성된 리드**로 한정.
- 매체·계정·캠페인·광고세트/그룹·소재·랜딩페이지·광고 리드·광고 리드에서 이어진 예약·귀속 결제/매출만 분석.
- 광고 출처가 없거나 식별되지 않는 데이터는 **채널이 아니라 데이터 품질(미귀속) 문제**(`광고 출처 확인 불가`)로 처리. `Unknown`을 정상 성과 채널로 집계하지 않음.
- 광고 외 유입을 추정해 특정 채널에 임의 귀속하지 않음. 전체 예약/매출을 매체별 성과로 임의 배분하지 않음.
- 광고와 연결 근거가 없는 결제는 **운영 매출**에는 포함하되 **마케팅 기여 매출·ROAS에는 포함하지 않음**.
- 본 문서에서 "채널"은 별도 언급이 없는 한 **유료 광고 매체**를 의미.
- 현재 운영하지 않는 매체의 신규 연동은 우선 개선 과제로 제안하지 않음.

**전제 전 흐름**: `광고비 집행 → 광고 클릭/랜딩 유입 → 리드 생성 → 연락·상담 → 예약 → 결제`.

## 1. Executive Summary

현 시스템은 **광고 성과 리포팅**은 갖췄으나, 두 핵심 사용자(리드 처리 실무자·다병원 관제 슈퍼어드민)의 실제 업무 흐름과 위 모집단 원칙에는 구조적 공백이 있다. 네 계통으로 요약한다.

**A. 헤드라인 수치를 신뢰할 수 없다 (P0).**
- 매출·광고비 집계 쿼리가 **1,000행에서 조용히 잘린다**. PostgREST 기본 상한(명시적 `.limit()` 없으면 1,000행, `lib/services/weeklyReport.ts:13` `SUPABASE_MAX_ROWS=1000`)을 다수 라우트가 우회하지 않는다. `dashboard/kpi/route.ts:31`(payments 무제한), `:29`(ad_campaign_stats 무제한), `:30`(leads `.limit(5000)`) — 한 KPI 카드 안에서 매출≤1,000행·리드≤5,000행·예약=정확(count) 세 상한이 섞인다. 월 결제 1,500건 병원은 매출 약 33% 과소, ROAS↓·CAC↑.
- 같은 이유로 `dashboard/channel`(payments `.limit(5000)`)과 `dashboard/kpi`(payments 무제한=1,000)의 매출 총합이 **서로 어긋난다**(채널 합이 총합 초과 가능).
- API 오류가 화면에서 **정상 0으로 위장**된다(`ads/stats/route.ts:59,112`가 에러를 `apiSuccess({stats:[]})`로 반환, `hooks/use-dashboard-data.ts:28-33,137-148`가 `res.ok` 미확인).

**B. 귀속·매출 정의가 모집단 원칙을 위반한다 (P0).**
- 메인 ROAS = **기간 전체 매출 ÷ 광고비**(주석 그대로 "리드 귀속 제한 없음", `dashboard/kpi/route.ts:63-64`). 기존 고객·워크인 매출까지 분자에 포함 → 광고 성과 과대. → 원칙 "전체 매출을 광고 성과로 해석 금지" 위반.
- 예약은 리드에 **FK로 연결되지 않는다**(`bookings`에 `lead_id` 없음). 요청하신 **Last Paid Touch Before Booking은 미구현**이고, 기본 귀속은 `customers.first_source` 퍼스트터치뿐(`attribution/summary/route.ts:28`), 귀속 윈도우(30/60/90일)도 없다.
- 매칭 실패 결제를 `Unknown` 채널에 **매출과 함께 집계**한다(`attribution/summary:153-156`, `dashboard/channel:116`). → 원칙 "Unknown을 정상 채널로 취급 금지" 위반.

**C. "오늘 할 일"과 "확인할 병원"을 보여주는 화면이 없다 (P1).**
- clinic_staff는 로그인 시 `/patients`(예약 캘린더)로 강제 이동(`app/(dashboard)/page.tsx:30-32`), clinic_admin은 `/`(성과 리포트)로. **둘 다 처리 대기 업무 큐가 아니다.** 리드 아웃바운딩 워크플로우는 `/campaigns`에 따로 있고, 캠페인을 하나 선택해야 상태 카운트가 보이며 `utm_campaign is null` 리드는 목록에서 제외된다(`api/campaigns/route.ts:145`). 9개 필수 업무 큐 중 **랜딩 화면에 실행 가능한 큐로 노출되는 것은 0개**.
- 슈퍼어드민 이상탐지는 `spend_spike`·`ctr_drop` **2종뿐**(`lib/ads-anomaly.ts`), 결과는 화면이 아닌 **SMS 요약 3줄**로만 나가고(fire-and-forget, 비영속) CPL 급등·전환 급락·지출 후 리드 0건·노쇼·동기화 중단·광고출처 미저장은 감지하지 않는다. 병원별 "확인 필요" 뷰가 없어 병원 미선택 시 전 병원이 **하나의 blended 합계**로만 보인다(`lib/api-middleware.ts:171-178`).

**D. 데이터 유실·누출·품질 공백 (P0/P1).**
- 상담 이력이 저장 때마다 **덮어써진다**(`consultations` upsert 1행/고객, `patients/[id]/consultation/route.ts:47-82`).
- agency_staff가 캠페인 목록을 열면 **배정 안 된 타 병원 데이터가 노출**된다(`api/campaigns/route.ts:149,155`가 `applyClinicFilter` 미적용).
- 광고 출처 식별률(매체·캠페인·소재별)·**귀속 실패 의심 매출**(§4.5)을 품질 지표로 측정하는 장치가 없다.

**최우선 레버리지 순서**: ① 집계 truncation·오류표시 수정(수치 신뢰 회복) → ② 실무자 업무 큐 + 슈퍼어드민 관제 뷰(대부분 기존 데이터로 가능) → ③ 광고 리드 식별·DQ 지표 + Last Paid Touch Before Booking 귀속(과거 근사 소급 + 신규 데이터부터 정확).

## 2. 사용자별 대시보드 제안

기존 화면을 유지하되 **랜딩 직후 "처리 대기 인박스 / 관제 뷰"를 최상단에 추가**하는 방식(전면 리뉴얼 아님). 모든 카드·알림은 클릭 시 **해당 조건으로 필터링된** 리드/예약 목록으로 이동해야 한다(현재는 클릭해도 무필터 전체 이동, `kpi-section.tsx:116`).

### 2.1 병원 실무자 대시보드 (업무 인박스)

clinic_staff 랜딩(`/patients`)·clinic_admin 랜딩(`/`) 상단에 공통 인박스 삽입. 각 큐 = 카운트 배지 + 클릭 시 필터 목록.

| 큐 | 정의(구현 쿼리) | 데이터 근거 | 지금 가능? |
|---|---|---|---|
| 신규 미처리 리드 | `lead_status='new'` | `idx_leads_lead_status` | ✅ 현재 데이터 |
| N시간 이상 미연락 | `lead_status='new' AND created_at < now()-Xh`(또는 `lead_notes` 0건) | 필드 존재 | ✅ |
| 오늘 재연락 대상 | `leads.next_contact_at = today` | **필드 없음** | ⛔ 신규 필드 |
| 연락 후 결과 미입력 | `contacted_at` 있고 결과상태 없음 | **필드 없음** | ⛔ 신규 필드 |
| 상담 후 미예약 | 여정단계 `consulted` & 비취소 예약 없음(`leads/page.tsx:40-55` 로직 재사용) | 존재 | ✅ |
| 오늘 예약 | `bookings.booking_datetime ∈ KST today`, `status≠cancelled` | 존재(현재 카드는 월누계 `patients:1221`) | ✅ |
| 오늘 취소·노쇼 | `bookings.status IN ('noshow','cancelled')` & today | 존재 | ✅ |
| 후속 조치 필요 고객 | 방문/결제 후 N일 경과·재방문 없음 | 파생 가능/플래그 권장 | △ |
| 최근 예약 전환 저하 | 광고 리드 기준 booking-rate 이동평균 대비 하락 | 계산 로직만 신규 | ✅ |

핵심 개선점:
- **리드 상태 화면 통합**: `lead_status`·`lead_notes`는 `/campaigns`에만 있고 "고객관리(`/leads`)"엔 없다(`leads/page.tsx`는 customers 쿼리). 실무자가 직관적으로 여는 화면에서 후속 업무가 안 보인다. `/leads`에 상태·메모·마지막활동 노출 또는 **캠페인 무관 "전체 광고 리드 인박스"** 신설.
- **광고 출처 확인 불가 리드도 작업 가능하게**: `utm_campaign` 결측 광고 리드(현재 캠페인 목록에서 제외)는 오가닉이 아니라 **DQ 경고가 붙은 광고 리드**다. 전체 리드 수·전체 CPL에는 포함하고 세부 매체 분석에서만 제외하며, DQ 큐로 노출해 실무자가 연락할 수 있게 한다.
- **"오늘" 스코프 고정**: `/patients` 요약 카드는 월누계(`patients:1432-1437`)라 매번 날짜필터 변경 필요 → 오늘 예약/노쇼/취소 전용 카드.

### 2.2 슈퍼어드민 대시보드 (관제 · 이상탐지)

병원 순위 나열 대신 **"확인 필요 병원만" 노출**. 현재 병원 미선택 시 blended 합계 하나(`api-middleware.ts:171-178`)여서 이상 병원을 못 찾는다. 아래 9개 질문(요청)에 1:1 대응하며, **모두 광고 모집단 기준**이다. 오가닉/직접/전화/추천 비교는 범위에서 제외.

| 질문 | 지표/판정(제안) | 현재 |
|---|---|---|
| CPL이 최근 기준보다 악화된 매체? | 매체 CPL vs 최근 N주 중앙값/요일평균, 표본 ≥K | ⛔(spend만) |
| 지출됐지만 리드 0건 캠페인? | `spend>임계 AND 광고리드=0` | ⛔ |
| 리드는 있는데 예약 전환 낮은 매체/캠페인? | 귀속 예약전환율 vs 기준 | ⛔ |
| 소재 리드품질/예약 전환 저하? | `utm_content`별 리드→귀속예약 전환 추이 | ⛔(소재 매출 라이프타임 버그 `creatives:88-94`) |
| 랜딩페이지 리드/전환 이상? | `landing_page_id`별 리드·전환 이상 | ⛔ |
| 광고 API vs 실제 리드 불일치? | `ad_campaign_stats`(클릭/스펜드) vs 광고 리드 수 정합 | ⛔(동기화 freshness 미추적) |
| 매체·캠페인·소재별 식별률이 떨어지는가? | 차원별 식별률 지표(부록 A) | ⛔ |
| 귀속 실패 의심 예약·매출이 급증하는가? | 귀속 실패 의심 지표(부록 A·§4.5) 추이 | ⛔ |
| 실제 기여 매체·캠페인·소재? | 코호트 ROAS 상위(§4) | △(blended·first-touch만) |

이상 판정 원칙(요청 반영):
- **비교 기준**: 직전 동일기간 + 최근 N주 이동평균/중앙값 + 요일평균(현재는 직전 ≤6일 단순평균, `ads-anomaly.ts:112-116`).
- **최소 표본**: 리드/전환 신호는 분모 표본 K 미만이면 억제(현재 광고 신호만 `history≥3`, `:107`).
- **경고 단계**: 단일 임계(2×/0.5×, `:118,133`) → warning/critical 2단계 + 지속일수 조건으로 일시 변동과 실제 이상 구분.
- **알림 4요소**: `문제 → 비교 기준 → 영향 범위 → 확인할 조치`(현재 메시지는 문제+기준만, 영향/조치 없음).
- **영속·화면화**: 현재 결과는 SMS 3줄 후 소멸(`ads-anomaly.ts:197-202`, `error-alert.ts` 인메모리 쿨다운은 콜드스타트마다 리셋). `anomalies` 테이블 + 관제 패널 필요.
- **코호트 성숙도 보정**: 최근 N일 유입 리드는 예약·결제 전환 시간이 부족하므로 **미성숙 코호트**로 표시하고 저성과 이상치 판정에서 제외(또는 별도 표시). D+0/D+7/D+30 성숙 곡선·매체별 평균 전환 소요기간 기준(§4.6).

## 3. 리드 업무 프로세스 분석

**추적 흐름**: 신규 → 연락 시도 → 상담/보류/거절 → 재연락 → 상담 성공 → 예약.

### 3.1 정보 구조(IA)의 근본 문제
- 워크플로우가 **직관과 반대로 배치**됨: 실제 리드 처리(상태 드롭다운 + 메모 타임라인 + 상태 필터)는 **"캠페인 리드"(`/campaigns`)**에 있고, "고객관리(`/leads`)"는 고객 중심(여정단계·등급·결제)이라 `lead_status`·메모·연락상태를 **전혀 표시하지 않는다**.
- `/campaigns`는 **캠페인을 하나 골라야** 상태 카운트가 보이고(`campaigns/page.tsx:906-928`), 목록 쿼리가 `utm_campaign is null`을 제외(`api/campaigns/route.ts:145`)해 **출처가 온전히 캡처되지 않은 광고 리드가 상태 화면에서 안 보인다** → 처리 누락 + DQ 은폐.

### 3.2 필드·이력 제공 여부 (근거 포함)

| 항목 | 상태 | 근거 |
|---|---|---|
| 고객명·전화번호 | ✅ | `campaigns/page.tsx:434,437` |
| 문의 시각 | ✅ = `leads.created_at` | `campaigns/page.tsx:447` |
| 문의 시술 | △ 1급 필드 없음 — `custom_data` 있을 때만 칩 표시. CSV "시술"은 `payments.treatment_name`(과거구매) | `lib/lead-custom-data.ts`; `leads/export/route.ts:177` |
| 연락 상태 | ✅ 단 `/campaigns`에만 | `campaigns/page.tsx:449-458` |

### 4.2 요청하신 6개 구분 — 판정
1. **건수 vs 고유 고객** — ❌ 혼용. 리드=행/이벤트(`kpi:45`), 예약은 KPI에선 행수(`kpi:32`)·퍼널에선 distinct 고객(`funnel:98-102`). 리드는 전화/고객 중복제거 없음.
2. **생성일 vs 실제 발생일** — ❌ 예약·방문·상담이 `created_at` 기준(`kpi:33`, `funnel:56,64`). "오늘 예약"=오늘 등록분. 매출·광고비만 실날짜.
3. **코호트 vs 기간 운영 전환율** — ❌ 전부 기간운영(`이벤트/이벤트`), 코호트·리드→예약 조인 전무.
4. **기간 전체 매출 vs 마케팅 기여 매출** — ❌ 메인 KPI는 전체매출(`kpi:46,64`). `/attribution`도 Unknown 포함.
5. **Blended vs 코호트 ROAS** — ❌ blended + first-touch 기간 ROAS만. 코호트 ROAS 없음.
6. **기존 고객 vs 광고 기여 매출** — ❌ 신규/재방문 미판독. 재방문·워크인 결제가 blended ROAS를 부풀림.

### 4.3 개정 KPI 정의 — 광고 모집단 기준

이 시스템의 신규 리드는 원칙적으로 유료 광고 유입이다. 따라서 UTM·캠페인 정보가 누락돼도 **광고 리드에서 제외하지 않는다.** 다음 3범주로 나눈다(부록 A).
- **전체 광고 리드**: 모든 광고 리드 → **전체 리드 수·전체 CPL 분모에 포함**(식별 실패 포함).
- **출처 식별 광고 리드**: 매체·캠페인·소재·랜딩 식별이 가능한 리드 → **세부 매체별 상세 분석에만 포함**.
- **출처 식별 실패 광고 리드**: 전체 성과에는 포함하되 **세부 매체에 임의 배분하지 않고** 데이터 품질 문제로 표시(부록 A).

식별 수준은 단계적으로 적용한다(부록 A): 매체 식별→매체별, 캠페인 식별→캠페인별, 소재 식별→소재별, 랜딩 식별→랜딩별, **세부 식별 불가→전체 성과에는 포함하고 해당 분석 차원에서만 제외**. `utm_campaign` 하나 결측이라고 리드 전체를 미확인 처리하지 않는다. **단위(건수/고유 고객)를 지표명·설명에 반드시 표시**한다.

| KPI | 개정 정의 | 단위 | 현재와의 차이 |
|---|---|---|---|
| 광고 리드 이벤트 수 | 활성 광고 매체 유입 리드 건수(식별 실패 포함) | **건수(이벤트)** | 광고 리드로 필터, 비광고만 제외(`kpi:45`) |
| 광고 유입 고유 고객 수 | 광고 리드의 distinct `customer_id` | **고유 고객** | 현재 미산출 |
| 전체 CPL | 전체 광고비 ÷ **전체 광고 리드 이벤트 수** | 건수 기준 | 식별 실패 리드도 분모 포함 |
| 매체별 CPL | 매체 광고비 ÷ 해당 매체 **식별** 리드 수 | 건수 기준 | 식별 리드만(세부 차원) |
| 예약 기여 리드 수 | Last-Paid-Touch로 예약에 귀속된 **리드 수** | **리드(이벤트)** | 신규 |
| 예약 고객 수 | 귀속 예약의 distinct `customer_id`(취소 제외) | **고유 고객** | 귀속 무관 전체 예약행 → 재정의(`kpi:32`) |
| 리드 예약 전환율 | 예약 기여 리드 수 ÷ 광고 리드 이벤트 수 · **분자·분모 모두 이벤트(리드)** | **메인 대시보드 권장** | 기간운영(`kpi:65`) → 코호트 |
| 고객 예약 전환율 | 예약 고객 수 ÷ 광고 유입 고유 고객 수 · **분자·분모 모두 고유 고객** | 고객 단위 | 신규 |
| 결제 고객 수 | 광고 귀속 예약 이후 결제 등록 distinct `customer_id` | 고유 고객 | 전체 결제 고객(`kpi:52`) → 귀속 제한 |
| 마케팅 기여 매출 | 광고 귀속 예약과 연결 가능한 결제 금액(취소·환불 차감) | 금액 | 전체 결제 합(`kpi:46`) → 귀속 결제 |
| 코호트 ROAS | **마케팅 기여 매출 ÷ 해당 광고 리드 코호트 광고비 × 100** | **% 또는 배수(금액 아님)** | **신규**(현재 없음) |
| 운영 매출 | 선택 기간 전체 결제 금액(광고 무관) | 금액 | 현 `kpi.totalRevenue` — **"운영 매출" 라벨, ROAS 해석 금지** |
| 노쇼율 | 노쇼 예약 ÷ (방문 + 노쇼) | % | 현재 미산출(`kpi` 부재) |

> **메인 대시보드 기본 전환 지표 = 리드 예약 전환율**(마케팅 운영 관점). 고객 단위가 필요하면 고객 예약 전환율을 병행하되, 카드·설명에 분모·분자 단위를 명시한다. **리드 이벤트 수와 고유 고객 수, 예약 건수와 예약 고객 수를 혼용하지 않는다.**

**두 매출·두 ROAS를 화면에서 분리**:
- **운영 매출 / 운영 현황** = 전체 결제(현 `dashboard/kpi.totalRevenue`) — 병원 매출 현황 카드로 유지, "광고 성과 아님" 문맥. **ROAS로 해석 금지.**
- **마케팅 기여 매출 / 코호트 ROAS** = Last-Paid-Touch 귀속 결제만. **코호트 ROAS = 마케팅 기여 매출 ÷ 코호트 광고비 × 100(% 또는 배수).** 채널·캠페인·소재 성과는 오직 이 값으로 산출.
- **광고 미귀속 매출**(= 운영 매출 − 마케팅 기여 매출)은 채널로 집계하지 않으며, 반드시 **정상 미귀속 운영 매출**과 **귀속 실패 의심 매출**로 분리한다(§4.5). 전체 차액을 오류로 처리하지 않는다.
- CAC/ARPC도 광고 귀속 결제 고객 기준으로 별도 산출(현 `kpi:54-59`는 전체 기준 → 운영 지표로만).

### 4.4 귀속 재정의 — Last Paid Touch Before Booking

| # | 규칙 | 구현 매핑 | 현재 데이터로 가능? |
|---|---|---|---|
| 1 | 동일 병원·동일 고객으로 리드↔예약 연결 | `bookings.customer_id + clinic_id → leads` | ✅ |
| 2 | 예약 **완료 최초 전환 시각** 이전 발생 **광고 리드만** 후보 | `leads.created_at ≤ 예약완료전환시각`(신규 `booking_confirmed_at`/`booked_at`, 과거는 `bookings.created_at` 근사) AND 광고 리드(부록 A) | ✅(과거 근사) |
| 3 | 전환 시점과 가장 가까운 광고 리드에 귀속 | 후보 중 `created_at` 최대 | ✅ |

| 9 | 매체뿐 아니라 캠페인·광고세트·소재·랜딩까지 귀속 | 리드 필드: `utm_campaign`/`campaign_id`, `utm_content`→`ad_stats.ad_id/ad_name`(소재), `landing_page_id`(랜딩). **광고세트/그룹은 리드 미저장** → `ad_stats` 매핑 보강(부분) | △ 대부분 가능 |
| 10 | 식별 불완전 리드는 임의 포함 금지, DQ 경고 | 부록 A 식별 규칙 | ✅ |
| 11 | 예약 일시 변경 시 기존 귀속 리드 **불변** | 재스케줄은 `booking_datetime`만 변경, `attributed_lead_id`·전환시각 고정 | ✅(신규 필드 후) |

**정확 귀속 불가 구간 처리**
- **현재 데이터로 소급 가능(과거 적용)**: 예약완료전환시각을 `bookings.created_at`로 **근사**해 `customer_id + created_at ≤ 예약(근사)시각`의 last-paid-touch를 조회만으로 산출·백필. 채널·캠페인·소재·랜딩까지 근사 귀속하되 **근사 귀속임을 표시**.
- **신규 필드(신규 데이터부터 정확)**: `bookings.booking_confirmed_at`(또는 `booked_at`, 예약완료 최초 전환 시각) + `bookings.attributed_lead_id`(FK), `payments.booking_id`(결제→예약→리드 사슬 확정), `payments.status/refunded_amount`(취소·환불 차감). 전환시각은 최초 확정 시 1회 stamping, 이후 재스케줄에도 불변.
- **광고세트/그룹**: 리드에 adset 식별자 없으면 정확 귀속 불가 → 소재·캠페인까지만 확정, 없는 구간은 "광고세트 미상" 표기(임의 배분 금지).
- **마이그레이션 범위**: 전환시각·FK 컬럼 추가 + 과거 예약 1회 역산 백필(근사). 소급은 근사, 신규는 stamping으로 정확화.

### 4.5 미귀속 매출 분리 — 정상 미귀속 vs 귀속 실패 의심

`운영 매출 − 마케팅 기여 매출`(광고 미귀속 매출)을 **전부 데이터 품질 오류로 처리하지 않는다.** 두 개념으로 분리한다.

| 구분 | 정의 | 처리 |
|---|---|---|
| **정상 미귀속 운영 매출** | 기존 고객, 귀속 기간(윈도우) 초과, 분석 코호트 이전 유입 등 **귀속 규칙상 정상 제외**되는 매출 | 운영 매출에 포함, 마케팅 기여 매출·코호트 ROAS 제외. **오류·이상 아님** |
| **귀속 실패 의심 매출** | 광고 리드·예약·결제 **연결 정보가 존재**하지만 데이터 누락/연결 오류로 귀속되지 않은 매출 | **DQ 경고·이상탐지 대상은 이것만.** 급증 시 §2.2 탐지 |

즉 데이터 품질 경고와 이상탐지는 **귀속 실패 의심 매출**에 한정하고, 정상 미귀속 매출을 오탐하지 않는다.

### 4.6 코호트 성숙도 (미성숙 코호트 오탐 방지)

최근 유입 리드는 예약·결제로 전환될 시간이 부족하므로 **신규 코호트를 즉시 저성과로 판단하지 않는다.**
- **성숙 지표**: D+0 리드 성과 → D+7 예약 전환 → D+30 결제·코호트 ROAS 를 단계로 관측.
- **미성숙 코호트 표시**: 관측 기간이 매체 평균 전환 소요기간에 미달한 코호트는 "미성숙"으로 명시하고 확정 성과로 표기하지 않음.
- **이상탐지 제외/별도**: 최근 N일 코호트는 저성과 이상치 판정에서 제외하거나 별도 표시(§2.2 코호트 성숙도 보정).
- **매체별 평균 전환 소요기간**: 리드→예약, 예약→결제 소요기간을 매체별로 산출해 성숙 기준선으로 사용.
- 현재 데이터로 D+0/D+7/D+30 곡선·소요기간은 `leads.created_at`·예약/결제 시각으로 산출 가능(신규 필드 불필요). 정확한 예약완료전환시각은 §4.4 신규 필드로 개선.

## 8. 단계별 로드맵

**Phase 0 — 수치 신뢰 회복 & 모집단 준수 (P0, 1~2주)**
B1(행 상한) → B4(오류 0 차단) → B2(테넌트 격리) → B3(상담 이력) → B5(운영매출/코호트 ROAS 분리·라벨) → B28(리드/고객 지표 분리) → B6(정상 미귀속 vs 귀속 실패 의심 분리). 데이터 변경 없이 조회/렌더/라벨 수정 중심. 완료 시 헤드라인 수치가 truncation·오탐·모집단 위반에서 벗어남.
*검증*: `npm run build`/`lint` + 결제 1,000건 초과 시나리오로 채널합=총합 확인.

**Phase 1 — 실무자 업무 큐 & 재연락 (P1, 2~4주)**
B8·B9·B16(기존 데이터로 6큐+오늘뷰+노쇼율+DQ 큐) → B10(재연락 신규 필드 3종) → B30(문자 알림 운영성) → B7·B17. 실무자가 "오늘 할 일"을 한 화면에서 처리, 재연락 누락 차단, 리드 첫 접점(SMS) 안정화.

**Phase 2 — 슈퍼어드민 관제·이상탐지 (P1, 3~5주)**
B11(광고 리드 식별·차원별 식별률) → B15(확인 필요 병원 뷰) → B13(9개 질문 이상탐지·화면화, 미성숙 코호트 보정) → B14(동기화 freshness). 병원 반복 확인 제거, 이상을 화면에서 조치.

**Phase 3 — 코호트 귀속 정합 (P1→P2, 4~6주, 과거 근사 + 신규 정확)**
B20·B23·B17(채널 정합 선정리) → B12(코호트 파이프라인) → B29(코호트 성숙도) → B18(Last Paid Touch, 예약완료전환시각 + FK·윈도우) → B19(환불 반영) → B24. 과거 소급 근사 → 신규 데이터부터 `booking_confirmed_at`·`attributed_lead_id` 정확화.

**Phase 4 — 구조 정리·완성도 (P2~P3)**
B21(상태 체계 분리·CHECK) → B22(목록 상태 유지) → B25(등급/recall) → B26·B27.

## 부록 A. 광고 리드 식별 규칙 (기존 필드, 신규 컬럼 없음)

**광고 리드 식별 수준(단계적)** — `utm_campaign` 하나가 결측이어도 리드 전체를 미확인 처리하지 않는다. 식별 가능한 차원까지만 상세 분석에 포함하고, 세부 식별 실패 리드도 **전체 광고 성과에는 포함**한다.

| 식별 수준 | 판정(기존 필드) | 분석 포함 범위 |
|---|---|---|
| 매체 식별 | `utm_source` 정규화 값이 활성 광고 매체와 매칭(`clinic_api_configs`(`is_active`)/`ad_campaign_stats.platform`, Meta/Google/Naver/TikTok/Dable) 또는 클릭ID(`fbclid`/`gclid`/`ttclid`, `lib/lead-custom-data.ts`가 표시에서만 제외·값 보존) | 매체별 성과 포함 |
| 캠페인 식별 | `utm_campaign`/`campaign_id` 존재 | 캠페인별 성과 포함 |
| 소재 식별 | `utm_content`→`ad_stats.ad_id/ad_name` 매핑 | 소재별 성과 포함 |
| 랜딩 식별 | `landing_page_id` 존재 | 랜딩페이지별 성과 포함 |
| 세부 식별 불가 | 위 차원 값 결측(예: `utm_campaign` 결측, `api/campaigns/route.ts:145` 제외 대상) | **전체 광고 성과에는 포함**, 해당 분석 차원에서만 제외 |

**3범주 집계 원칙**(§4.3): **전체 광고 리드**(모든 광고 리드 → 전체 리드 수·전체 CPL 분모 포함) / **출처 식별 광고 리드**(매체·캠페인·소재·랜딩 상세 분석 포함) / **출처 식별 실패 광고 리드**(전체 성과 포함, 세부 매체에 임의 배분 금지·DQ 표시). 오가닉/직접 추정 분류 금지.

**데이터 품질(DQ) 지표(현재 데이터로 산출)** — 하나로 묶지 않고 **차원별 식별률**로 분리:
- **매체 식별률** = 매체 식별 리드 ÷ 전체 광고 리드
- **캠페인 식별률** = 캠페인 식별 리드 ÷ 전체 광고 리드
- **소재 식별률** = 소재 식별 리드 ÷ 전체 광고 리드
- **랜딩 식별률** = 랜딩 식별 리드 ÷ 전체 광고 리드
- **귀속 실패 의심 예약·매출**(§4.5) = 연결 정보가 있으나 미귀속된 건수·금액(**정상 미귀속과 구분**)

각 식별률 급락·귀속 실패 의심 급증은 §2.2 이상탐지 대상.

## 부록 B. 상태 체계 분리 (리드 처리 상태 vs 고객 여정 단계)

현재 두 병렬 체계가 개념적으로 혼용된다.
- **리드 처리 상태** = `leads.lead_status`(campaigns 소유). 값: `new/no_answer/consulting/consulted(legacy)/booked/hold/rejected`, **CHECK 제약 없음**(`20260316_lead_status.sql`).
- **고객 여정 단계** = `bookings.status` + `consultations.status` + `payments`로 **읽기 시점 파생**(leads/patients 소유), 저장 안 됨(`leads/page.tsx:40-55`).

혼용 지점: `booked`/`예약완료`/`방문완료` 라벨이 두 체계에 중복. `lead_status='booked'` 시 예약 자동 INSERT(`leads/[id]/route.ts:51-87`)하지만 여정 측은 `bookings.status`로 판정 → 두 "예약" 카운트가 상충. 상담 UI가 API 화이트리스트에 없는 `상담대기` 제공 → 400.

제안(기존 구조 활용): (DB) `lead_status` CHECK·값 정규화, 기반 테이블 스키마 migration화. (API) 상태 화이트리스트를 `lib/security.ts`로 일원화, `booked→bookings` 브리지 idempotent + 링크. (UI) "리드 상태"(campaigns) vs "여정 단계"(leads/patients) 라벨 분리, 여정 배지는 공용 `lib/journey.ts`에서 렌더.

1. KPI 정의서(§4.3)와 백로그(B5)의 코호트 ROAS 계산식 일치 — `기여매출 ÷ 코호트광고비 × 100` ✅
